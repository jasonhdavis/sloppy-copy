# Task 008.5: Pipeline Verification

## Objective
Verify the end-to-end flow and error handling of the orchestrator.

## Context
The orchestrator is the main entry point for automated runs.

## Verification Steps

### 1. Full Run
Run:
```bash
python manage.py run_pipeline
```
**Expected Output:** Success message with counts for the four stages:

```
Ingested: <int>
Evaluated: <int>
Matched: <int>
Rewritten: <int>
```

These counts must correspond to the rows processed in each stage of the pipeline.

### 2. Run Tracking
Check the `PipelineRun` table in the admin.

Model reference: [`PipelineRun`](orchestrator/models.py:1)

Verify the following fields:

```
started_at
completed_at
status
stories_ingested
stories_evaluated
stories_assigned
rewrites_generated
error_log
metadata
```

Expected state after a successful run:

```
status="completed"
completed_at != NULL
```

### 3. Partial Run
Run:
```bash
python manage.py run_pipeline --mode=evaluate
```
Verify only evaluation steps were performed.

Expected behavior:

```
stories_evaluated > 0
stories_ingested unchanged
stories_assigned unchanged
rewrites_generated unchanged
```

Database verification example:

```python
from orchestrator.models import PipelineRun

run = PipelineRun.objects.latest("started_at")
print(run.status, run.stories_evaluated)
```

## Success Criteria
- [ ] End-to-end run completes without exceptions.
- [ ] `PipelineRun.status == "completed"`.
- [ ] `stories_ingested`, `stories_evaluated`, `stories_assigned`, and `rewrites_generated` match database counts.
- [ ] At least one `Rewrite` row exists after a successful run.
- [ ] Logs show clear transitions between pipeline stages.

Database verification queries:

```python
from stories.models import Story
from rewrites.models import Rewrite

Story.objects.filter(status="rewritten").count()
Rewrite.objects.count()
```
